背景导读

过去三个月，Code Agent 相关的 RL 工作集中在两个问题上：第一，怎样把奖励从“最终测试是否通过”扩展到更可靠的执行、验证、数据合成与多阶段代理反馈；第二，怎样把训练系统从单轮代码生成迁移到多轮工具调用、延迟奖励、sandbox 执行和 test-time verification 的组合体。

这批论文里，最值得警惕的是指标证据并不均匀。部分工作给出明确数字，例如 AgentV-RL 的 +25.2%、RECRL 的 Pass@1 +1.23%-5.62%、COVERT 的 BFCL/ACEBench 改善；另一部分只给定性描述。本报告在所有 abstract 未给量化数字的条目上显示“abs 未给量化数字”，避免用前端模板补造默认百分数。

OPD 方向在本窗口高度收敛为 On-Policy Distillation：让 student 在自己生成的轨迹上接收 teacher 的稠密 token 级监督。Off-Policy Distillation 仅作为反义基线或子组件出现；Online Preference、Optimal Policy、Output Policy 等候选含义在本窗口没有被检索到独立原文。

Exploring Pass-Rate Reward in Reinforcement Learning for Code Generation

作者

Xin-Ye Li, Ren-Biao Liu, Yun-Ji Zhang, Hui Sun, Zheng Xie, Ming Li

机构

abs 未直接列出；作者署名风格类似南京大学 LAMDA 团队，未在 abs 中确认

关键贡献
  1. 系统比较 pass-rate reward 与 binary pass-all-tests reward 在 critic-free RL（GRPO / RLOO）上的差异。
  2. 实证发现 pass-rate reward 虽缓解奖励稀疏，却没有可靠地优于 binary，这是对代码 RL 常见 dense reward 假设的反直觉负面结果。
  3. 给出梯度方向分析：部分通过的解会产生相互抵消的策略梯度，从而把概率质量拉离真正正确解。

方法亮点跨 base model 与算法做大规模消融；提出 miscalibrated surrogate 概念解释 dense reward 失效。报告在页面中以 reward design 橙色标签呈现，并把它作为 RL 方向的反例锚点。

指标增益N/A；负面结果论文，作者刻意不主张提升数字。

Term clarification

OPD 在本窗单义指 On-Policy Distillation

本报告中的 OPD 仅指 On-Policy Distillation，即 student 在自己采样出的轨迹或状态上接收 teacher 的稠密监督。这一点由 2604.00626 综述、2604.13016 对应仓库 thunlp/OPD，以及本窗 9 篇相关 arXiv 工作共同支撑。

Off-Policy Distillation 在这些文献中是对照基线或 recipe 的 cold-start 子步骤，不作为 OPD 的独立活跃含义。页面中的 OPD 卡片因此按效率、稳定性、机理、代理化、多教师、RLVR 杂交等主题组织。

  3. abs 称在数学 / 科学 reasoning benchmark 上持续优于 vanilla OPD，但没有给具体数字。

方法亮点把 OPD 与 RL 方差降低工具直接连起来，是理论化 OPD 的稳定性补充。

指标增益abs 未给量化数字；仅有定性优于 vanilla OPD 的表述。

Synthesis

跨方向观察：RL × OPD 的交叉点与争议

1. 轨迹归属正在成为核心变量

RL 方向强调 rollout 服务化、agentic verifier、tool execution；OPD 方向强调 student-visited states。二者都在远离静态数据集范式，转向“模型实际会走到哪里，就在那里学习或验证”。

2. 稠密信号并不自动更好

Pass-rate reward 的负面结果与 OPD 的 failure mode 诊断相互呼应：更密的 token 或测试信号可能带来偏差、污染或梯度抵消，必须配合支撑集匹配、步骤重加权或 verifier 可信度估计。

3. Code agent 的评测仍不统一

RL 工作横跨 Codeforces、SWE-Bench Verified、BFCL、ACEBench、OR benchmarks、TikZ 程序合成；OPD 工作则多用 reasoning、AIME、agentic 和 code generation 混合指标。跨论文排序容易误导。

Open questions

三个开放问题

Verifier 的可信度如何校准？

RL 论文越来越依赖执行器、工具调用、solver、judge 或 agentic verifier。下一步不仅要验证答案，还要估计 verifier 自身在分布外任务上的错误模式，尤其是工具输出噪声和 sandbox 差异。

OPD 与 RLVR 能否共享 credit assignment？

OPD 提供 token 级密集监督，RLVR 提供方向稳定的环境信号。Self-Distilled RLVR 与 SOD 指向一种混合路线，但如何在多步 code agent 轨迹里分配 step-level 和 token-level 权重仍未统一。

哪些指标能真正代表 code agent 能力？

Codeforces、SWE-Bench、LiveCodeBench、工具调用基准和程序合成基准测量的是不同能力。报告不做全行业 SOTA 排名，因为多数 abstract 没有公开可比数字，且任务分布差异会放大误读。
