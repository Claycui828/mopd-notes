§1 · Motivation

为什么需要 OPD

现代大模型后训练经常遇到「跷跷板」问题：针对数学的 RLVR 会缩短推理轨迹，并可能损害开放写作；RLHF 能提升偏好对齐却可能牺牲严格指令跟随；工具使用 RL 又可能让模型偏离 STEM 基准。每一个专门化阶段都可能以牺牲其他能力为代价，最终发布一个「什么都能保住」的统一模型变得很困难。

在线策略蒸馏（On-Policy Distillation, OPD）提供了一种常用修复路径：让学生模型先采样自己的轨迹，再在这些 rollout 上匹配教师分布，通常使用 reverse KL。它带来密集的 token 级监督，并几乎无缝嵌入 GRPO 风格训练循环。

自然扩展是多教师 OPD（MOPD）：为每种能力选最强 checkpoint 当教师，让学生在同一阶段吸收多教师能力。教师常与学生共享 tokenizer 与模型谱系，工程接入成本相对较低。

§2 · Background

从 GRPO 到 OPD

2.1 原始 GRPO 目标

GRPO 要最小化的损失可以写为：

GRPO loss L_GRPO(θ) = − 𝔼[ min( ri,t · Âi,t, clip(ri,t, 1−ε, 1+ε) · Âi,t ) ]

其中，token 级 PPO 式重要性采样比率为：

Importance ratio ri,t(θ) = π_θ(yi,t | x, yi,<t) / π_θ_old(yi,t | x, yi,<t)

组相对优势（group-relative advantage）定义为：

Group-relative advantage Âi,t = ( Ri − mean(R1, …, RG) ) / std(R1, …, RG)

这里 Ri 是一组 rollout 中第 i 条轨迹的 outcome reward。

注

原文为说明清晰使用原始 GRPO 目标。实际训练中常见的改法（优势不除以标准差、不同长度归一化方式等）都可以套用到后续目标中。

2.2 用 reverse KL 替换优势项

OPD 可以看成把 GRPO 里的优势项替换为学生策略与教师策略之间的 reverse KL 信号：

OPD advantage (reverse-KL) Âi,t = sg[ log π_infer(yi,t | x, yi,<t; θ_teacher) / π_train(yi,t | x, yi,<t; θ) ]

其中 sg(·) 是 stop-gradient 操作。由于优势来自教师分布而非组内 baseline，不需要每个 prompt 采多条 rollout来计算相对优势。把 group size 设为 1 既合理也更利于吞吐。

直观上：教师对某个学生采样 token 给出的概率高于学生时，优势为正，梯度推动学生上调；反之下调。教师相当于一个密集的 token 级奖励。

2.3 为什么是 reverse KL？

简短结论

forward KL 让学生覆盖教师所有模式，结果更平滑但不聚焦；reverse KL 让学生集中在教师主模式上，结果更尖锐，有时也更自信。对真实文本生成这种多模态分布，reverse KL 的 mode-seeking 行为通常是更稳妥的默认选择。

reverse KL 自然适配 OPD，因为它允许从学生策略采样：

Reverse KL KL( π_train , π_infer ) = 𝔼[ log π_train / π_infer ]

forward KL 是离线序列知识蒸馏的常见目标。它更偏 mean-seeking，会迫使学生覆盖教师所有模式：简单任务也许 OK，但真实生成模式很多，forward KL 容易让学生在低密度区域分配过高概率，生成很不可能的样本。

Figure — forward vs reverse KL 行为对比 mean-seeking（forward）vs mode-seeking（reverse）的概率分布示意 · 原文 image

reverse KL 更偏 mode-seeking：鼓励学生在教师主模式上分配高概率，在低密度区域分配低概率。因此 OPD 通常采用 reverse KL。

2.4 训练与推理 gap：IcePop

训练引擎和推理引擎即使输入相同，也可能因 kernel、batch invariance 等因素产生不同 logits。MoE RL 中专家路由的非确定性会进一步放大差异。IcePop 的做法：如果 train/infer 概率比落在容忍区间 [α, β] 之外，就 mask 掉对应 token，直接放弃最嘈杂的更新。

IcePop · masked OPD/GRPO L_IcePop(θ) = − 𝔼[ M( p_train / p_infer ; α, β ) · min( ri,t·Âi,t, clip(ri,t, 1−ε, 1+ε)·Âi,t ) ]

其中 M(·; α, β) 是 mask 函数。原文指出，后面讨论的四个模型中，MiMo-V2-Flash、GLM-5、Nemotron-Cascade 2 都明确采用了 IcePop。

§4 · Final-stage approaches

MiMo-V2-Flash 与 GLM-5

这两者都把 OPD 放在后训练最后一步用作整合，但教师构成与损失增强方式都不同。

MiMo-V2-Flash
混合来源 · 教师 + Self · ORM 叠加

从 general SFT checkpoint 出发，教师池来自 pipeline 不同位置：领域 SFT 模型、RL 专家模型，以及学生自身快照（Self）。最后阶段同时承担合并异构教师能力与缓解过专门化两件事。

  • 每个 prompt 带领域标签，确定性选单一教师
  • OPD 信号上叠加 outcome-reward advantage
GLM-5
同谱系 · 阶段末 checkpoint · 纯 reverse-KL

从 Reasoning → Agentic → General RL pipeline 的最终 post-RL checkpoint 出发，把每个前序阶段的 terminal checkpoint 当教师。Prompt 来自各教师原本的 RL 训练集——routing 是隐式的。

  • 教师共享同一谱系，只是取自 pipeline 不同时点
  • 更像「最终能力恢复」，纯 reverse-KL
Self-distillation（MiMo）

这里的 Self 是 MOPD 开始时的学生快照——一个固定、稳定的参考分布。当 SFT/RL 教师把学生推向不熟悉区域时，向 Self 蒸馏可以防止灾难性漂移。

Teacher routing（MiMo）

每个 prompt 带领域标签，确定性选择单一教师。token 级优势只针对这个教师计算。多教师聚合是通过样本级领域混合隐式完成的，而不是 token 级 ensemble。

两者的差异具体落在两点：

  • 教师构成不同：MiMo 按 prompt 领域路由到 SFT、RL specialist 或 Self；GLM-5 用每个前序阶段的 terminal checkpoint。
  • 损失增强方式不同：MiMo 在 reverse KL 上叠加 scaled outcome-reward advantage；GLM-5 使用纯 reverse-KL OPD。
§5 · Beyond imitation

Beyond Teachers · 如何超过教师

纯 OPD 的局限：学生被优化为匹配教师 token 级分布，而非直接优化任务成功率。强教师的密集监督是优势，但纯模仿也会传播教师错误、风格偏差和次优推理模式。

MiMo-V2-Flash 的做法是把 OPD advantage 与 ORM advantage 组合：

Combined advantage Âi,t = ÂOPDi,t + α · ÂORMi,t

其中 α 控制 ORM advantage 相对 OPD log-ratio 的权重。论文未报告其具体值或敏感性，因此 dense imitation 与 outcome supervision 的 trade-off 仍然隐含在实现里。

关键符号：

ÂORM由 outcome reward model 计算的优势
G > 1每 prompt 多 rollout（GRPO baseline 需要）
G = 1每 prompt 单 rollout（纯 OPD 足以）
Open question

论文把 ORM 项泛称为 "advantages computed by the ORMs, including GRPO"。如果用 GRPO，就需要每个 prompt 采多条 rollout 估计组 baseline——这些 rollout 是否也用于 OPD log-ratio？还是 OPD 单独使用单条 rollout？论文没有说明。

谁是最好的教师

在 12 个 benchmark 任务中，RL、Self、SFT 分别在 6 / 5 / 1 个任务上是最佳教师。模式比较清楚：

  • RL 教师主导数学、代码、推理这类可验证奖励 benchmark
  • Self 在更开放、宽泛的任务上更易胜出，因为 SFT/RL 教师可能破坏校准

MOPD 学生在 12 个任务中有 8 个超过最佳教师，最大提升来自 Arena-Hard Hard Prompt（+4.1）；也有 4 个落后，最大损失来自 BrowseComp（−6.3，该任务 SFT 是最佳教师）。

原文提醒

论文并没有明确说最终 MOPD 使用了每个 benchmark 上单独表现最好的教师。更重要的是，最佳独立教师不等于最佳蒸馏教师——OPD 关心的是教师在学生 on-policy 样本上的条件分布，而不只是最终任务准确率。

原文还把它和 ExOPD 联系起来：ExOPD 不是叠加 outcome reward，而是引入额外 reference model，测量 teacher-reference delta，沿教师相对 reference 改进的方向做外推。二者结构相似——保留 OPD 作为 dense imitation core，再加一个 scaled corrective term；不同点在于信任的纠偏信号是可验证奖励，还是教师相对参考模型的提升方向。

§6 · Mid-stage stabilizer

Nemotron-Cascade 2

关键观察：随着训练环境增加，能力会漂移。例如 non-math RLVR 会降低模型 entropy、缩短推理轨迹，从而影响数学；偏 RLHF 的优化可能牺牲指令跟随。

MiMo 和 GLM-5 把 MOPD 放在末尾整合；Nemotron-Cascade 2 把它插入中间当稳定器——不是只在最后恢复能力，而是在专门化阶段之间周期性重新锚定模型。

6.1 MOPD 前的 Multi-Domain RL

先做一个 Multi-domain RL 阶段，覆盖三类能力：

STEM MCQ · 55%
Tool calling · 30%
IF · 15%
STEM 多选问答 agentic tool calling 结构化指令跟随

选择把这些 domain 混进一个 RL 阶段，原因有两个：(a) 混合训练没有观察到 benchmark 退化，反而在 MMLU-Pro、τ-Bench、IF-Bench 上提升；(b) 这些数据集响应长度与验证耗时相近，能减少长生成或慢环境验证带来的训练低效。

6.2 三教师配置

math teacher
SFT initialization 本身——团队认为高质量 SFT 数据已提供足够数学信号；进一步 math RL 反而可能缩短推理轨迹。
RLHF teacher
来自 RLHF 阶段的 helpfulness / safety alignment checkpoint。
multi-domain teacher
IF-RL + Multi-domain RL checkpoint，覆盖 agentic 与结构化指令跟随能力。

教师 checkpoint 直接从 Cascade RL pipeline 中按 benchmark 类别选最强验证 checkpoint，便于构建能力多样的教师池而不引入外部模型家族。Prompt 来自 RLHF / IF-RL / Multi-domain RL 数据池及 AceReason-Math。同样使用 IcePop 风格的 truncated importance weighting 处理 train-inference mismatch。

6.3 训练效率收益

MOPD 在样本效率与步数效率上都更高：从同一初始 checkpoint 出发，它在 AIME25 上明显快于 GRPO 达到教师级性能；在 ArenaHard v2 上用更少 step 超过 RLHF。

同一起点下，MOPD 比纯 GRPO 更快达到教师级 AIME25 表现，并以更少 step 在 ArenaHard v2 上超过 RLHF。

§7 · Engineering at scale

DeepSeek-V4 · 用基础设施放大 MOPD

独立训练领域专家，每个专家使用领域 SFT + RL（GRPO）。其他系统的 domain teachers 往往混合 SFT、RL、Self；DeepSeek-V4 更强调大规模 RL specialists。OPD 主要在两个维度上放大：

  • 全词表 logit 蒸馏：保留完整 logit 分布计算 reverse KL，而不仅在 sampled token 上估计。
  • 教师数量：超过 10 个教师共同蒸馏一个学生，覆盖数学、编码、agent、IF 至少四类领域；部分领域按 Non-think / Think High / Think Max 三种 reasoning effort 进一步拆分。

叠加 1.6T 参数 / 49B activated（DeepSeek-V4-Pro）与 1M token 上下文，对训练基础设施提出新要求。

7.1 高效教师调度（FP4 / FP8 混合精度）

主要内存挑战有两个：(a) 教师数量多且每个都可能很大；(b) 全 logits 形状 num_tokens × vocab_size，长序列下朴素存储会爆炸。

DeepSeek-V4 的旁支优化是用 FP4（MXFP4）量化做 inference-only forward。它适用于教师与 reference 的前向推理；训练 step 仍保持 FP8，并通过无损 FP4→FP8 dequantization 保持反向 pipeline 不变。每个 batch 可能触发 10+ 教师前向，推理侧节省非常关键。

7.2 容错 rollout · token 级 WAL

DeepSeek 使用 preemptive scheduler，运行任务可能被抢占；大规模硬件故障也常见。普通监督训练中失败 mini-batch 常可重试；但 OPD/RL rollout 更敏感——如果恢复时重新用新随机数采样，就不是继续同一条 on-policy 轨迹，而是在采新样本。

这会带来长度偏差：

  • 短 completion 更容易在中断前完成
  • 长 completion 更容易被中断
  • 中断样本被从头重采时，长样本被替换概率更高
  • 最终 accepted sample 分布偏向短输出

因此从头重新生成在数学上不正确。DeepSeek 的方案是 token-granular Write-Ahead Log（WAL）：每生成一个 token 即刻追加到该请求的 WAL；抢占时暂停并保存未完成请求的 KV cache；恢复时从持久化 WAL 与 KV cache 继续 decode；如遇 fatal hardware error，则用 WAL token 重新 prefill 重建 KV cache。

Note

如果 inference stack batch-invariant 且确定性，用同一 seed 做 deterministic regeneration 也能保证正确性。但它仍要从头 rerun decoding，效率不如 WAL。

对 agentic 任务，rollout 还包含环境转移。如果环境转移不可复现，OPD 数据会被破坏（学生可能被训练在原始环境状态下不会发生的轨迹上）。DeepSeek 的 DSec 为每个 sandbox 维护全局有序 trajectory log，记录命令调用与结果；恢复时重放缓存结果而不是重新执行命令——这是 token WAL 在 sandbox 场景下的对应物。

7.3 全局规划 · metadata 与 per-token payload 解耦

OPD 同时需要全局规划与 token 级训练数据。全局规划关心 shuffle、变长 pack、教师分配；这些任务不需要沉重的 token 级 payload。在 full-vocab OPD + 1M context 下，朴素加载会带来巨大内存压力。

类别 用途 轻 / 重 内容
Metadata Planning 轻量 sample id · length · teacher id · domain · offsets · packing metadata
Per-token payload Training 重型 tokens · attention masks · loss masks · logprobs · teacher hidden states · rewards / advantages

具体做法：全量 rollout dataset 只加载 metadata，用来做全局 shuffle 与 packing layout 计算；重型字段通过 shared-memory data loader 读取，避免节点内重复拷贝，并在 mini-batch 消费后立即释放。

§8 · Synthesis

总结

四份报告覆盖的模型差异很大：参数规模 309B → 1.6T，上下文 32k → 1M，能力偏向从 agent-first 到 math-first 不等。但它们都收敛到相似的 OPD 核心。

A · 收敛点

共享同一个 primitive

  • 以学生 rollout 上的 reverse KL 作为核心损失
  • 都采用多教师框架
  • 四中其三明确采用 IcePop 风格 mismatch 缓解

动机各异——MiMo / DSv4 偏能力合并；GLM-5 / Nemotron 偏遗忘恢复——但 reverse-KL OPD 对两类问题都成了合适 primitive。

B · 分歧点

四个轴上的差异

  • 教师构成：MiMo 混合 / GLM-5 同谱系阶段末 / Nemotron 三类 / DSv4 10+ specialists
  • Pipeline 位置：三者末端整合，Nemotron 居中稳定
  • OPD 信号增强：MiMo 叠加 ORM；其余纯 OPD
  • 工程规模：仅 DSv4 推到必须专门设计基础设施
C · 未来方向

开放问题

  • 扩展教师数量与规模：边际收益是否持续
  • 黑盒蒸馏：仅有教师采样 token、无 logits 时如何做
  • 教师-学生共同演化：蒸馏后再做 specialist training，外层循环成本/收益
